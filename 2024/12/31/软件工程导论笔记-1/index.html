2.软件工程的概念与发展历程

1.历程

  • 第一阶段：程序设计阶段
    • 时间：计算机诞生后(1946)
    • 特点：
      • 软件工作：代码编写
      • 软件评估：程序=数据结构+算法、编程技巧
  • 第二阶段：软件工程阶段
    • 时间：大容量、高速度计算机出现，NL、OS、DB出现
    • 特点：上述情况导致了软件危机，催生了软件工程
      • 软件工作：
        • 代码编写+需求分析、测试、维护——软件工程
        • 系统化、规范、可度量的工程化方法应用于软件
      • 软件评估：
        • 可读性、可理解性、可测试性和易修改性

2.需求工程概述

1.需求的定义

定义：

需求就是以一种清晰、简洁、一致、无二义性的方式，对一个待开发软件系统中各个有意义功能和性能等方面的陈述的集合

需求的基本性质

  • 必要性
  • 无歧义性
  • 可测试性
  • 可跟踪性
  • 可测量性

2.需求的意义

  • 需求体现软件价值、决定软件命运

    • 需求本质反映的是如何为客户和用户等利益相关者拟定新的目标组织前景

    • 错误与缺乏创新的需求将导致软件的失败

  • 需求是软件项目和产品的根源，是后续设计、实现、测试、维护等所有活动的依据

3.需求的层次

  • 业务需求
    • 对应前景与范围文档
  • 用户需求
    • 对应用户需求文档(用例文档)
  • 系统需求
    • 对应软件需求规格说明文档(系统需求规格SRS)

C

4.需求的类型

  • 功能性需求
    • 解释：是对系统应该提供的服务、系统应该如何响应特定的输入、系统在特定的情形中应该如何表现等的陈述
    • 在某些情况下，功能性需求还可以明确的陈述系统**”不应该做什么”**
  • 非功能性需求
    • 业务规则
      • 公司政策、政府法规、工业标准、计算算法……
    • 质量属性
      • 外部质量
        • 可靠性
          • 成熟性、容错性、可恢复性
        • 易用性
          • 易理解性、易学性、易操作性、吸引性
        • 效率/性能
          • 时间特性、资源利用、安全性
      • 内部质量
        • 可维护性
          • 易分析性、易改变行、稳定性、易测试性
        • 可移植性
          • 适应性、易安装行、共存性、替换行
    • 外部接口
      • 系统接口：描述一个应用如何与系统的其他应用交互
      • 用户接口：规约了软件产品与用户之间接口的逻辑特性
      • 硬件接口：系统所要求的支持和协议类型
      • 软件接口：允许与其他软件产品交互
      • 内存约束：易失性存储和永久性存储的特性和限制
      • 操作：规约用户如何使系统进入正常和异常的运行，以及在系统正常和异常运行下如何与系统进行交互
      • 地点需求：描述系统安装以及如何调整一个地点，以适应新的系统
    • 设计约束
      • 限制系统或系统构件的设计方案
      • 设计约束本身，对其进行权衡调整很困难，甚至不可能
      • 示例：
        • 系统必须用C++或其他面向对象语言编写
        • 特定应用消耗算力平均不超过50%

3.需求获取与分析

需求螺旋迭代过程的第一部分

1.基本路径与方法

  • 用户调研
    • 基本方法包括
    • 用户访谈
    • 调查问卷
    • 用户反馈

2.面临的挑战

  • 需求的模糊性和隐蔽性
  • 需求的多样性
  • 需求的领域专业性
  • 需求的易变性

3.获取与分析方法

1.用户调研

目的是解决需求的模糊性(说不清楚要什么)

最有效的方法：

  • 原型法

    • 用户调研+原型法——解决需求的模糊性

    • 原型既可以是可操作的软件界面，也可以是草图原型等

    • 根据运用原型的目的与方式不同，可分为：

      • 快速原型

      • 进化原型

2.深入观察与挖掘

目的是解决需求的隐蔽性(客户告知的需求是表象/客户不愿告知/客户未知)

方法包括：

  • 现场观察与挖掘
  • 数据分析
  • 市场行业分析
  • 竞品分析
  • 头脑风暴
  • 群体化方法

最有效的方法(理论)：

  • 扎根蘑菇群理论
  • 黄金圈理论
    • 都属于现场观察与挖掘方法

3.用例

目的是解决需求的多样性

方法：

  • 针对不同用户的不同需求，回答：
    • 系统应该为每个用户做什么
      • 从每个用户case或story进行各项需求评判
    • 有效界定系统范围，为整个用户提供最大价值

步骤：

  • 识别User类型
  • Case/Story构建(为所有用户)
  • 汇总、协商确定与优先级排序

4.需求文档编写

需求螺旋迭代过程的第二部分

1.需求规格说明的作用与要求

软件需求规格说明，简称SRS，也称为功能规格说明、产品规格说明、需求文档

1.作用

  • 记录共识
  • 确认共识
  • 完整正确地传递共识给开发人员

2.撰写要求

撰写要求需要符合文档的软工原则

  • 完整性
    • 三个需求层次的需求都要在SRS中有对应的文档信息
    • 应涵盖需求两大类型
    • 涵盖不同用户需求
  • 一致性
    • 同一文档的需求一致
    • 不同用户的需求不相互矛盾
    • 不同层次需求的一致性
  • 可阅读性
    • 长度适当，版面规整，条理清楚，层次分明
  • 无二义性
    • 字面意思
  • 可修改性
    • 能依据该需求成功设计测试用例
  • 可跟踪性
  • 可测试性
  • 可维护性

2.需求规格说明的撰写方法

一般性方法：套用模板法

步骤：

  • 第一步：选择文档模板
    • 行业内认可度高的模板/企业已有模板
  • 第二步：裁剪文档模板
  • 第三步：选择描述语言，进行文档写作
    • 满足SRS的一般性要求
    • 描述语言包括
      • 自然语言
      • 结构化语言(例如标准化的表格模板)
      • 图形化的建模语言(如UML)

1.业务需求规格说明

  • 模板：项目前景与范围文档
  • 来源：微软开发过程/统一过程
  • 描述语言：
    • 自然语言
    • 结构化自然语言(如标准化的表格模板)

2.用户需求规格说明

  • 模板：用例文档
  • 来源：统一过程RUP的模板
  • 描述语言：
    • 自然语言
    • 结构化自然语言
    • 图形化建模语言——UML

3.系统需求规格说明

  • 模板：系统需求规格说明(系统SRS)
  • 来源：IEEE 830-1998
  • 描述语言
    • 自然语言
    • 结构化自然语言(如标准化的表格模板)
    • 图形(如用户界面原型图)

4.小结

SRS的高效撰写

  • 方法：套用模板+UML可视化建模
  • 对模板内容的深入理解
    • 三个层次需求文档的依赖关系

5.需求验证与变更

1.需求验证

1.意义

需求验证的重要性：如果后续开发或系统投入使用时才发现需求文档中的错误，那么返工的代价更大

2.任务

验证需求文档，发现错误/伪需求并修改

  • 有效性检查
  • 一致性检查
  • 完备性检查
  • 现实性检查/可行性分析

3.方法

  • 需求评审
    • 正式技术评审、非正式技术评审
    • 参与评审人员：用户/客户、开发人员、测试人员、文档编写人员，项目经理等，7人左右
  • 原型法
    • 确定合适原型
    • 将需求验证涉及的业务过程或场景定义出来
    • 原型法与用例法结合
  • 需求验证漏斗
    • 依据黄金圈法则

  • 需求强调评估
    • 需求强度=潜在用户数+使用频率
  • 编写测试用例
  • 编写用户手册
  • 自动的一致性分析(如CASE工具)

2.需求签约

1.定义

需求验证结束后的重要工作——需求签约

  • 软件需求签约，是软件项目最关键的一步
  • 强调签约的必要性、权威性、严肃性、实用性是软件成功的基本保证

2.意义与内容

意义：

  • 对需求分析不断沟通完善的过程和过程产生的结果SRS的确认
  • 签约一旦生效，表明：
    • 同意了这份文档代表此刻对项目需求的最佳理解
    • 被确认的SRS是软件需求的基线，是后续软件开发和验收的依据
  • 对未来可能出现的需求变化的应对指南
    • 进一步变更将在此基线的基础上，依照项目的变更过程进行
    • 变更批准后，重新协商项目成本、资源、进度的约定

6.统一过程

统一过程，全称Rational Unified Process，即RUP

基于UML的过程框架

1.概述

  • 提出者：Rational公司+UML的创立者
  • RUP内容及特点
    • 一种典型完整的软件过程模式
      • 生命周期模型(过程模型)
      • 人员
      • 方法
      • 产品
    • 一种软件过程产品
      • Rational/IBM开发维护
      • 与Rational一系列其他软件开发工具集成

2.RUP软件过程模型

1.概要

迭代与增量的生命周期结构

  • 横轴
    • 时间组织
    • 体现动态结构
      • 生命周期中的动态结构——四个阶段
    • 术语：阶段、里程碑、迭代和增量
      • 四个阶段：先启阶段，精化阶段，构建阶段，产品化阶段

  • 纵轴
    • 内容组织为逻辑活动
    • 体现静态结构
      • 生命周期中的静态结构——9个核心工作流程
    • 术语：工作流程、活动、角色、工件
      • 工作流程代表了所有角色、活动与工件的逻辑分组情况，即软件过程模式中的3个要素

2.核心工作流程

生命周期中的静态结构——9个核心工作流程

核心工作流程组成：

  • 核心过程工作流程：前6个
    • 业务建模
    • 需求
    • 分析设计
    • 实施
    • 测试
    • 部署
  • 核心支持工作流程：后3个
    • 配置与变更管理
    • 项目管理
    • 环境

表示方法：

  • UML中：协同图、时序图、活动图
  • RUP中：活动图

内容：

  • 一套完整的UML全色i官方使用指南
1.业务建模

意义：

  • 打造软件核心竞争力
  • 不同的业务建模决定软件不同的命运

启示：

  • 创新能力
  • 具备不同行业专业领域的专业知识
2.需求

捕获需求，并使开发人员与客户就该需求达成共识

使用用例方法——对需要的功能和约束进行提取、组织、文档化，理解系统所解决问题的定义和范围

3.分析设计

将需求转化为未来系统的设计

以架构设计为核心——产品的适应性、可扩展性

7.配置与变更管理

描述如何再多个成员组成的项目中控制大量的产物，并提供了准则来管理演化系统中的多个变体，跟踪软件创建过程中的版本

8.项目管理

平衡竞争的目标、管理风险、克服各种约束，从而成功交付使用户满意的产品

9.环境

向软件开发组织提供软件开发的环境，包括过程和CASE工具

3.四个阶段

生命周期中的动态结构——四个阶段

  • 每个阶段：由1次或多次迭代完成
  • 迭代过程是受控的：
    • 项目计划中：制定
      • 迭代的次数
      • 每次迭代的持续时间
      • 每次迭代的目标
    • 详细迭代计划中：制定
      • 具体的工作流程
1.先启阶段
  • 目标：建立业务用例、确定项目的范围和边界
  • 结束里程碑：生命周期目标里程碑
  • 评审标准：
2.精化阶段
  • 目标：建立稳定的架构、编址项目计划、淘汰项目中最高风险元素
  • 结束里程碑：生命周期架构里程碑
  • 评审标准：
3.构建阶段
  • 目标：所有构件和应用程序功能被开发并集成为产品、所有功能被详尽的测试
  • 结束里程碑：最初操作性能里程碑
  • 评审标准：
4.产品化阶段
  • 目标：将软件产品交付给用户群体
  • 结束里程碑：产品发布里程碑
  • 评审标准：

3.RUP与螺旋模型异同

相似点：

  • 迭代与增量：都采用迭代和增量式开发
  • 风险管理：高度重视风险管理

不同点：

  • 生命周期的结构：
    • RUP的生命周期模型分了4个阶段
    • 螺旋模型采用螺旋式的周期结构
  • 开发进程的管理方式：
    • RUP采用相对严格的迭代流程，一个迭代周期内，多个工作流程可以并行
    • 螺旋模型强调串行的活动和连续的反馈
  • 关注的重点：
    • RUP重视过程和质量管理
    • 螺旋模型重视风险管理
  • 适用范围与灵活性
    • RUP适合中到大型的软件开发项目，特别适用于有多个团队协作的复杂项目
    • 螺旋模型适用于风险较高的项目，特别是在需求不确定或项目规模较大时

7.敏捷过程

敏捷过程，全称Agile Process，即AP，中文名又称敏捷开发/敏捷实践

1.概述

  • AP的提出

    • 互联网广泛应用后
    • 为了矫正繁琐的软件过程
    • 2001年2月正式发布敏捷软件宣言

  • AP内容及特点

    • 软件开发宣言
      • 使用AP的软件团队具有快速工作、快速响应变化的能力
    • 4条基本价值观+12条原则
      • 过程模型+人员+方法+产品及关系

  • AP旗下的敏捷型软件过程流派

    • 重点讲述SCRUM和极限编程XP(eXtreme Programming)

  • AP的掌握与应用
    • 思想方式较为前卫
    • 宣言定义的只是偏好，并不是说完全排斥某些东西，而是更重视一方

2.4条价值观

  • 个体和交互胜过过程和工具
    • 人是最重要因素
    • 当然不用过程和工具也很蠢
  • 可以工作的软件胜过面面俱到的文档
    • 软件开发的主要目标，是交付给用户可以工作的软件
    • 过多的面面俱到文档比过少的文档更糟
      • 耗费时间精力，甚至与代码不同步时产生误导
  • 客户合作胜过合同谈判
    • 客户不可能一次性把需求全搞清楚发给乙方
  • 响应变化胜过遵循计划
    • 变化是事实
    • 响应变化胜过遵循计划的意义在于——变更代表机遇

4.极限编程

极限编程，全称eXtreme Programming，即XP

  • 敏捷过程模型+方法技术

1.过程模型

2.技术方法

提出了14项最佳实践(技术方法)

  • 用户素材与用户故事

    • 用户素材
      • 正在进行的关于需求谈话的助记符
      • 一个计划工具，客户使用它并根据它的优先级和估算代价来安排实现该需求的时间
    • 用户故事(User Story)
      • 将故事分解为任务用于规划系统迭代
      • 类似于RUP的User Case

  • 结对编程

    • 两个开发人员，同一台电脑，一个编程，一个保证代码正确性可读性
    • 作用：非正式的同级评审-代码复审
    • 注意：结对编程是加强开发人员沟通评审的一种方式，但不唯一

  • 测试驱动开发

    • XP强调测试先行
      • 编码前，先把测试写好
    • 测试自动化是测试先行的必要条件
    • 与RUP相比：
      • RUP和XP对测试都很重视，但是两者对于测试在整个项目开发周期内首先出现的位置的处理不同

  • 重构

    • 调整、优化系统内部结构，减少复杂性、消除冗余、增加灵活性、提高性能
    • 运用方法：
      • 代码重构与简单化设计思想一脉相承
      • 注意不要过分依赖重构，而轻视设计
    • 对比RUP：RUP以构架为中心

5.SCRUM

1.概述

  • 发展史：
    • 1980，两个人在IBM项目合作中诞生SCRUM
  • 内容：
    • 包括一系列实践的过程框架和预定义角色
  • 目的：
    • 一定程度上提供敏捷项目进展状况的外部可见性
  • 影响力：
    • 被PLOP作为组织和过程模式的标准

2.过程模型

SCRUM过程模型也是一种迭代式增量的过程

  • SCRUM的生命周期是不停循环的
  • 整个生命周期在固定的时间段内完成，称为Sprint
    • Sprint通常需要2-4周

1.计划和体系结构设计

2.Sprint
  • Sprint周期在2-4周
  • Spring产出是可用的、潜在可发布的产品增量
  • Sprint迭代周期长度一旦确定就保持不变
  • Sprint进行过程中，开发目标、质量验收标准、团队构成不再变化
